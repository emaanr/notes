# Introduction

- Recap of the process to build executable:

<p align="center" width="100%">
    <img src="img/figure-1.png">
</p>

- `.h` + `.c` $\rightarrow$ `.c` $\rightarrow$ `.s` $\rightarrow$ `.o` $\rightarrow$ `.exe` + `.a` $\rightarrow$ `.exe` & `.so`
  1. Preprocessor uses `.h` headers to perform textual insertion in `.c` files which produces larger `.c` files after textual insertion is complete.
  2. Compiler takes these `.c` C files and turns them into `.s` assembly code files.
  3. Assembler transforms `.s` assembly code files into `.o` object files.
  4. Linker links `.o` object files and `.a` external libraries to create executable.
  5. Dynamic Linking phase puts `.exe` executable in memory alongside any relevant `.so` shared objects.

# Preprocesser

`.h` + `.c` $\rightarrow$ `.c`

- Textual replacement.
- Think of the preprocessor as something that knows how to take string in the C program and replace them with _other_ strings.
- Not as complicated as it may seem.
  - Sees a preprocessor and just replaces it with text, regardless of validity of the code.
- Can be done with `gcc -E`.
- After this step we have big ugly `.c` files that no one wants to looks at.

## Why?

- Allows the ability to do some "Generics" since C does not necessarily allow this.
  - "Poor Man's Generics"
- Some incorrectly assume we can use the preprocessor to reduce function call overhead.

## `#include`

- String directive that indicates where and what textual replacement needs to occur.

## `#define`

- User-defined Macros.
- Preprocessor knows to do a textual replacement defined by the user.
- Opertaion is not happening, text is being copy pasted in-place of the macro.
  - This simplifies code but is an easy way to make bugs if you don't understand that the addition operation is not done by the preprocesser.

### Examples

1. `#define ADD(x, y) ((x) + (y))`
2. `#define DIVO(x) ((x)/0)`

# Compiler

- `.c` $\rightarrow$ `.s`
- Takes aforementioned "big ugly `.c` files" generated by the preprocessor and transforms them into assembly `.s`.

## Resolves

- Local variabls
- Field names in structs
  - Since field names are just offsets.
  - Example: `foo.bar`.
- Function params/args

# Assembler

- `.s` $\rightarrow$ `.o`
- Takes assembly files and transforms them into object files.

## Resolves

- Relative branches if any.
- Most symbol resolution occurs in compiler.

## Relocatable Object Files

- Object files created at this point.
- We still have:
  1. Global variables
  2. Function names
- The linker will handle these.

## ELF

- "Executable and Linkable Format" in Linux/Unix.
  - Used to be `a.out`.
    - Still exists as a default name for calling `gcc` despite the file itself being an `ELF` while the historically `a.out` was a separate type of file entirely.
- Goal of this format is to have all metadata necessary to resolve the rest of, in other words "link", the symbols.

### Anatomy of an ELF

```
ELF:  header (ELF)
      program header
          .text
          .data
          .bss
          .symtab
          .rel.text
          .rel.data
          .debug
      section header
```
